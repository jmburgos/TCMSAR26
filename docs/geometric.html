Working with geometries in sf objects

library(tidyverse)
library(sf)

The sf package provides a large set of functions to operate on the geometries of sf objects. This chapter covers the most useful ones, organised by what they do:

  • Validity: checking and fixing geometry problems.
  • Generators: creating new geometries from existing ones (buffers, centroids, hulls, grids, type casting, set operations).
  • Simplification: reducing geometric complexity.
  • Measurements: areas, lengths, and distances.
  • Aggregation: dissolving and combining features.
  • Attribute-geometry relationships: understanding what happens to attributes when geometries change.

We will use the Nephrops functional units as our main example dataset throughout, projected to World Mercator for accurate distance and area calculations:

nfu <- read_sf("./data/nephrops_fu.gpkg") |>
  st_transform(3395)
ImportantAlways use projected data for geometric operations

Geometric operations (areas, distances, buffers) require planar (Cartesian) coordinates. Geographic coordinates (latitude/longitude) will produce wrong results — areas in square degrees, distances in degrees, and distorted buffers. Use st_is_longlat(x) to check, and st_transform(x, crs) to reproject. See the CRS chapter for guidance on choosing an appropriate projection.

Validity

Real-world spatial data frequently contains geometry errors: self-intersecting polygons, duplicate vertices, rings that don’t close properly. These cause cryptic errors in spatial operations. It is good practice to check validity before doing any analysis.

st_is_valid(nfu)        # Is each geometry valid?
 [1] TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE
[16] TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE
st_is_simple(nfu)       # Is each geometry simple (no self-intersections)?
 [1] TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE
[16] TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE TRUE
st_is_empty(nfu)        # Are any geometries empty?
 [1] FALSE FALSE FALSE FALSE FALSE FALSE FALSE FALSE FALSE FALSE FALSE FALSE
[13] FALSE FALSE FALSE FALSE FALSE FALSE FALSE FALSE FALSE FALSE FALSE FALSE
[25] FALSE FALSE FALSE FALSE FALSE FALSE

If geometries are invalid, st_make_valid() attempts to repair them automatically:

nfu_fixed <- st_make_valid(nfu)
all(st_is_valid(nfu_fixed))
[1] TRUE
Tip

It is good practice to run st_make_valid() on any spatial data you read from an external source before doing operations on it, especially if the data comes from a shapefile or an older GIS system.

Generating new geometries

Sampling

st_sample() generates random points within polygons or along linestrings. The result is an sfc object (a geometry column without attributes). Use st_as_sf() to convert it to a full sf object if you need to attach data.

set.seed(100)

# Random points within polygons
pts1 <- st_sample(nfu, size = 100, type = "random")

# Random points along the boundary of polygons
nfu_lines <- st_cast(nfu, "MULTILINESTRING")
pts2 <- st_sample(nfu_lines, size = 100, type = "random")

ggplot() +
  geom_sf(data = nfu, fill = "grey90") +
  geom_sf(data = pts1, colour = "red", size = 0.8) +
  geom_sf(data = pts2, colour = "blue", size = 0.8) +
  labs(title = "Random points inside (red) and on boundary of (blue) polygons") +
  theme_bw()

Centroids

st_centroid() returns the geometric centre of each feature. For irregular or concave polygons the centroid can fall outside the polygon itself. st_point_on_surface() guarantees a point that is always inside (or on the boundary of) the polygon, which is often more useful for labelling.

centr  <- st_centroid(nfu)
centr2 <- st_point_on_surface(nfu)

ggplot() +
  geom_sf(data = nfu, fill = "grey90") +
  geom_sf(data = centr,  colour = "red",  size = 2) +
  geom_sf(data = centr2, colour = "blue", size = 2, shape = 17) +
  labs(title = "Centroids (red circles) vs point on surface (blue triangles)") +
  theme_bw()

Buffers

st_buffer() creates a zone around a geometry at a given distance (in the units of the CRS — metres for most projected CRS). It works on points, lines, and polygons.

# Buffer around points: creates circles
buf_pts <- st_buffer(pts1, dist = 100000)  # 100 km

ggplot() +
  geom_sf(data = buf_pts, linetype = "dashed", fill = NA) +
  geom_sf(data = pts1, colour = "red") +
  theme_bw()
# Buffer around lines: creates a corridor
buf_lines <- st_buffer(nfu_lines, dist = 50000)  # 50 km either side

ggplot() +
  geom_sf(data = buf_lines, aes(fill = name), alpha = 0.5) +
  geom_sf(data = nfu, fill = NA, colour = "black") +
  theme_bw()
# Buffer around polygons: expands the polygon outward
buf_poly <- st_buffer(nfu, dist = 20000)  # 20 km outward

ggplot() +
  geom_sf(data = buf_poly, aes(fill = name), alpha = 0.4) +
  geom_sf(data = nfu, fill = NA, colour = "black") +
  theme_bw()
# Negative buffers shrink a polygon inward (only valid for polygons)
buf_neg <- st_buffer(nfu, dist = -20000)  # 20 km inward

ggplot() +
  geom_sf(data = nfu, fill = "grey90") +
  geom_sf(data = buf_neg, aes(fill = name), alpha = 0.6) +
  theme_bw()

Convex and concave hulls

A convex hull is the smallest convex polygon that encloses a set of points — think of stretching a rubber band around the outermost points. To get a single hull for a set of points, first union them into a MULTIPOINT with st_union(), then apply st_convex_hull().

pts1_u <- st_union(pts1)        # Combine into a single MULTIPOINT
chull  <- st_convex_hull(pts1_u)

ggplot() +
  geom_sf(data = chull, fill = "lightblue", alpha = 0.5) +
  geom_sf(data = pts1, colour = "red", size = 0.8) +
  theme_bw()
Note

st_union() is also used more generally to dissolve internal boundaries between polygons (merging multiple polygons into one) and to cluster individual points into a MULTIPOINT geometry. Without calling st_union() first, st_convex_hull() would return a degenerate “hull” for each individual point.

A concave hull fits more tightly around a point cloud, following the actual shape of the data rather than just the outermost extent. Modern sf (version 1.0-14 and later) includes st_concave_hull() directly:

# ratio controls how tight the hull is (0 = very tight, 1 = convex hull)
conc1 <- st_concave_hull(pts1_u, ratio = 0.1)
conc2 <- st_concave_hull(pts1_u, ratio = 0.5)

ggplot() +
  geom_sf(data = chull,  colour = "black", fill = NA, linetype = "dashed") +
  geom_sf(data = conc2, colour = "blue",  fill = NA) +
  geom_sf(data = conc1, colour = "red",   fill = NA) +
  geom_sf(data = pts1, size = 0.8) +
  labs(title = "Convex hull (dashed), concave ratio=0.5 (blue), ratio=0.1 (red)") +
  theme_bw()

Grids

st_make_grid() creates rectangular or hexagonal grids over the extent of an sf object. Hexagonal grids are increasingly popular for spatial analysis because each cell has equal distance to all its neighbours.

sq_grid  <- st_make_grid(nfu, n = c(10, 15))
hex_grid <- st_make_grid(nfu, n = c(10, 15), square = FALSE)

ggplot() +
  geom_sf(data = sq_grid,  fill = NA, colour = "grey60") +
  geom_sf(data = hex_grid, fill = NA, colour = "red", alpha = 0.5) +
  geom_sf(data = nfu, fill = NA, colour = "black") +
  theme_bw()

To get just the bounding box of an sf object as a polygon, use st_as_sfc() on the bounding box:

bbox_poly <- st_as_sfc(st_bbox(nfu))

ggplot() +
  geom_sf(data = bbox_poly, fill = NA, colour = "red", linetype = "dashed") +
  geom_sf(data = nfu, fill = "grey90") +
  theme_bw()

Geometry type casting

st_cast() converts geometries from one type to another. Common uses include converting POLYGON to LINESTRING (to get the boundary as a line), splitting MULTIPOLYGON into individual POLYGONs, or combining POINTs into a LINESTRING for a track.

nfu_lines <- st_cast(nfu, "MULTILINESTRING")

WarningCasting from MULTI to single geometry types

When casting from MULTIPOLYGON to POLYGON (or MULTILINESTRING to LINESTRING, etc.), each part of a multi-geometry becomes a separate row. Attribute values are duplicated across all resulting rows. Be careful when doing this before area or length calculations, as you may end up double-counting.

Set operations on geometries

These binary operations take two sf objects and return new geometries based on their spatial relationship. They are among the most powerful tools in sf.

# Load a second polygon layer to work with
ices_er <- read_sf("./data/ices_ecoregions.gpkg") |>
  st_transform(3395)

# Select just a couple of areas to keep examples clear
north_sea <- ices_er |> filter(ecoregion == "Greater North Sea")
celtic_sea <- ices_er |> filter(ecoregion == "Celtic Seas")

st_intersection() returns the overlapping part of two geometries. This is equivalent to a spatial “AND” — give me everything that is in both A and B.

# Which Nephrops functional units fall within the Greater North Sea ecoregion?
nfu_north_sea <- st_intersection(nfu, north_sea)

ggplot() +
  geom_sf(data = north_sea, fill = "lightblue", alpha = 0.3) +
  geom_sf(data = nfu_north_sea, aes(fill = name)) +
  theme_bw() +
  labs(title = "NFUs intersecting the Greater North Sea ecoregion")

st_difference() returns the part of the first geometry that does not overlap with the second — spatial “A minus B”.

# Parts of the Greater North Sea that are NOT covered by any NFU
north_sea_no_nfu <- st_difference(north_sea, st_union(nfu))

ggplot() +
  geom_sf(data = north_sea, fill = "grey90") +
  geom_sf(data = north_sea_no_nfu, fill = "red", alpha = 0.5) +
  theme_bw() +
  labs(title = "Parts of the Greater North Sea not covered by any NFU")

st_union() combines geometries, dissolving internal boundaries. Applied to a whole layer it returns a single merged polygon.

# Dissolve all NFUs into a single polygon
nfu_dissolved <- st_union(nfu)

ggplot() +
  geom_sf(data = nfu_dissolved, fill = "lightgreen") +
  theme_bw() +
  labs(title = "All NFUs dissolved into one polygon")

st_sym_difference() returns everything that is in either A or B but not in both — the spatial “XOR”.

NoteA note on CRS in set operations

Both objects in a binary set operation must share the same CRS. sf will throw an error if they do not. Always check with st_crs(a) == st_crs(b) or transform one to match the other before proceeding.

Simplification

Complex geometries with many vertices are slow to plot, slow to process, and large on disk. Simplification reduces the number of vertices while preserving the overall shape.

st_simplify() uses the Douglas-Peucker algorithm. The dTolerance argument sets the maximum allowable deviation from the original shape in CRS units (metres for projected data).

helcom <- read_sf("./data/helcom.gpkg") |>
  st_transform(3395)

helcom_simple <- st_simplify(helcom, dTolerance = 5000)  # 5 km tolerance

# Compare sizes
format(object.size(helcom),        units = "Kb")
[1] "1466.5 Kb"
format(object.size(helcom_simple), units = "Kb")
[1] "82.9 Kb"
p1 <- ggplot() + geom_sf(data = helcom)        + labs(title = "Original") + theme_bw()
p2 <- ggplot() + geom_sf(data = helcom_simple) + labs(title = "Simplified (5 km)") + theme_bw()

library(patchwork)
p1 + p2

The rmapshaper package offers ms_simplify(), which uses the Visvalingam-Whyatt algorithm. This often produces cleaner results than Douglas-Peucker for geographic data, particularly for coastlines, as it avoids the “spiking” artifacts that st_simplify() can introduce at high levels of simplification.

library(rmapshaper)
helcom_ms <- ms_simplify(helcom, keep = 0.05)  # Keep 5% of vertices

ggplot() + geom_sf(data = helcom_ms)

Aggregation

Aggregation in sf combines features by a grouping variable, summarising both attributes and geometries. This is done with group_by() and summarise(), exactly as in dplyr, with sf handling the geometry automatically.

Dissolving polygon boundaries

A common task is merging polygons that share a grouping variable, dissolving the internal boundaries between them.

ices_er <- read_sf("./data/ices_ecoregions.gpkg") |>
  st_transform(3395) |>
  mutate(zone = case_when(
    ecoregion %in% c("Arctic Ocean", "Barents Sea") ~ "Arctic & Subarctic",
    ecoregion %in% c("Bay of Biscay and the Iberian Coast",
                     "Mediterranean Sea", "Black Sea") ~ "Mediterranean & Black Sea",
    TRUE ~ "North Atlantic"
  ))

ices_zones <- ices_er |> 
  group_by(zone) |> 
  summarise(area_km2 = sum(area_km2)) |>   # do_union = TRUE is the default
  st_transform(4326) # For easier plotting

ggplot() +
  geom_sf(data = ices_zones, aes(fill = zone), alpha = 0.7) +
  coord_sf(ylim=c(30,85)) +
  theme_bw()
Note

The default do_union = TRUE in summarise() uses st_union() to dissolve internal boundaries between merged polygons. This is what you want for polygon aggregation. Using do_union = FALSE would keep internal boundaries and can produce invalid geometries.

Combining sf objects by row

To stack two sf objects with the same columns and CRS, use bind_rows() (tidyverse) or rbind() (base R):

sm_ices_er <- ices_er |> filter(area_km2 <  500)
lg_ices_er <- ices_er |> filter(area_km2 >= 500)

all_ices_er <- bind_rows(sm_ices_er, lg_ices_er)
nrow(all_ices_er) == nrow(ices_er)  # Should be TRUE
[1] TRUE

Attribute-geometry relationships

When we change geometries, we need to think carefully about whether the associated attributes still make sense. This is one of the subtler aspects of working with spatial data.

Consider a worked example:

  • We have a polygon representing a survey area, with an attribute total_cod giving the total cod caught in that area.
  • We split the polygon into three sub-areas by depth zone.
  • sf copies the total_cod attribute to all three sub-polygons — but now each one claims the entire survey total. The attribute is no longer meaningful.

The sf package formalises this through attribute-geometry relationships (AGR):

AGR type Meaning Examples
constant The attribute is the same everywhere within the geometry Bottom type, depth class
aggregate The attribute is a summary over the whole geometry Total abundance, mean temperature
identity The attribute uniquely identifies the entire geometry EEZ name, ecoregion name

By default sf does not know which type applies to each column, and warns you when doing operations that might make attributes inconsistent. You can suppress these warnings by setting the AGR explicitly:

st_agr(ices_er)  # Currently NA for all columns
ecoregion  area_km2      zone 
     <NA>      <NA>      <NA> 
Levels: constant aggregate identity
st_agr(ices_er) <- c(
  ecoregion = "identity",
  area_km2  = "aggregate",
  zone      = "constant"
)

st_agr(ices_er)  # Now set
ecoregion  area_km2      zone 
 identity aggregate  constant 
Levels: constant aggregate identity
# sf now knows that 'ecoregion' identifies the whole polygon,
# so it won't carry it over unchanged to a sub-geometry
ices_ct <- st_centroid(ices_er)
st_agr(ices_ct)  # 'identity' becomes NA for the centroid
ecoregion  area_km2      zone 
 constant aggregate  constant 
Levels: constant aggregate identity

In practice, setting AGR is more useful for suppressing false-positive warnings than for catching real errors — but understanding the concept helps you reason about whether your attribute values still make sense after geometric operations.
